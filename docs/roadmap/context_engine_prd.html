§01

Product Overview

The Context Engine is a standalone infrastructure component that sits between a document retrieval system and an LLM generation call. It implements context engineering — the discipline of deciding what information enters the model's context window, how much of it, in what order, and at what level of compression — as an explicit, observable, configurable software layer.

Without a context engine, LLM systems operating under multi-turn, multi-document, token-constrained conditions degrade predictably: relevant documents are dropped, conversation history accumulates noise, prompts overflow silently, and model performance collapses at thresholds that cannot be predicted in advance.

🔎
Hybrid Retrieval

Adaptive-alpha blending of dense embeddings and TF-IDF with query-type routing. Best of both paradigms, automatically.

🏆
Two-Phase Re-ranking

Heuristic pre-filter + cross-encoder scoring for principled document ordering at any scale.

🧠
Decay-Based Memory

Cognitive-inspired exponential decay with importance scoring, deduplication, and durable cross-session persistence.

✂️
Semantic Compression

Embedding-aware sentence scoring with original-order restoration. Preserves meaning, not just characters.

📊
Token Budget Enforcement

Slot-based allocator with disciplined reservation order. Context never overflows silently again.

🔭
Full Observability

Context trace logging, token audit, quality evaluation, and A/B experiment infrastructure built in.

§02

Problem Statement

The Core Failure Mode

Naive RAG systems fail within the first few multi-turn interactions. The failure is not in retrieval quality — it is in the complete absence of architectural control over what enters the context window. The consequences are deterministic and well-documented:

Failure ModeMechanismObserved Impact
Context OverflowRetrieved documents + history exceed token budget; LLM call fails or truncates silentlyImmediate failure; unpredictable output
Context RotIrrelevant content accumulates; attention budget diluted across noise95% → 60% accuracy at unpredictable length (Chroma, 2025)
Memory BloatAll conversation history retained without decay; old turns crowd out current relevance39% performance drop from fragmented multi-turn context
Retrieval NoiseNear-duplicate documents fill top-K; genuinely relevant but lexically distant documents excludedCritical documents dropped; model reasons from noise
StatelessnessNo cross-session persistence; context rebuilding wastes tokens and degrades experienceEnterprise unusable; continuity impossible
⚠ Critical Finding

Every model tested in the Chroma 2025 study of 18 frontier LLMs degraded as input grew. The degradation is non-linear and model-specific, making it impossible to predict when a system will break without an explicit token budget enforcement mechanism.

What Existing Solutions Miss

  • RAG frameworks (LangChain, LlamaIndex) do retrieval but not context assembly discipline
  • Prompt engineering focuses on instruction quality, not information flow control
  • Vector databases provide storage and retrieval but no compression or budget enforcement
  • LLM APIs have no context management — they accept tokens, they don't curate them
  • Memory libraries (Mem0, Zep) provide persistence but not integrated pipeline orchestration

No existing off-the-shelf component addresses context engineering as a complete, integrated pipeline. The Context Engine fills this gap.

§03

Goals & Non-Goals

Goals

GOAL-01Must Have
Guarantee that context packets assembled for LLM calls never exceed the configured token budget, under any combination of document count, history depth, or query type.
GOAL-02Must Have
Provide measurably higher retrieval relevance than TF-IDF-only or embedding-only retrieval, validated by NDCG@5 on a domain-specific evaluation set.
GOAL-03Must Have
Implement exponential memory decay with configurable parameters, ensuring high-importance turns survive longer than low-importance turns without requiring manual annotation.
GOAL-04Must Have
Produce a full context trace for every build() call — a serialized, queryable record of every allocation decision, compression ratio, and document retention outcome.
GOAL-05Should Have
Persist memory state across sessions using a configurable backend (SQLite for single-user; PostgreSQL for multi-user), with full API compatibility between backends.
GOAL-06Could Have
Expose the entire pipeline as an MCP server, enabling integration with LangGraph, AutoGen, Semantic Kernel, and any MCP-compatible orchestration framework.

Non-Goals

  • This is NOT an LLM. The Context Engine assembles context; it does not generate responses.
  • This is NOT a vector database. It integrates with one; it does not replace one.
  • This is NOT a prompt engineering library. Prompt construction is a separate concern.
  • This is NOT a fine-tuning pipeline. Knowledge is retrieved, not baked into weights.
  • This is NOT a UI or chat interface. It is infrastructure, consumed via API.

§06

Component: Hybrid Retriever

Functional Requirements

RET-01Must
Support three retrieval modes: keyword (BM25/token matching), tfidf (TF-IDF + cosine), and hybrid (adaptive-alpha blend).
RET-02Must
Implement a QueryClassifier that selects alpha dynamically based on query type: keyword-heavy (α ≈ 0.35–0.45), balanced (α ≈ 0.60–0.70), semantic/conversational (α ≈ 0.75–0.85).
Rationale: Fixed α=0.65 is domain-dependent and requires manual tuning. Adaptive routing removes a fragile configuration knob.
RET-03Must
Implement an LRU embedding cache (configurable size, default 10,000 entries). Cache hit must return in <2ms; cache miss triggers fresh embedding computation.
RET-04Should
Expose a pluggable embedding provider interface: LocalTransformer, OpenAIEmbeddings, CohereEmbeddings. Default to sentence-transformers/all-MiniLM-L6-v2.
RET-05Must
Degrade gracefully when sentence-transformers is unavailable: fall back to TF-IDF with a logged warning. Random embeddings are NOT acceptable in production mode.

Performance Targets

<1msKeyword mode
<5msTF-IDF mode
<30msHybrid (CPU cached)
<10msHybrid (GPU)
§07

Component: Two-Phase Re-ranker

Re-ranking determines the final document order before compression. A two-phase design delivers cross-encoder accuracy at heuristic speed for typical document counts.

Phase 1 — Heuristic Pre-filter

RNK-01Must
Score each candidate document using: final_score = retrieval_score × w₁ + tag_importance × w₂ + freshness_bonus × w₃. Default weights: 0.65 / 0.25 / 0.10.
RNK-02Must
Tag importance values and domain tag sets must be configurable per deployment. Default tag set and importance multipliers must be documented and overridable via config.

Phase 2 — Cross-Encoder Scoring

RNK-03Should
When document count exceeds a configurable threshold (default: 20), invoke a cross-encoder model (default: cross-encoder/ms-marco-MiniLM-L-6-v2) to score each query-document pair from the Phase 1 top-20.
Rationale: Cross-encoder is 1 model call per document. At 20 documents it adds ~40–80ms CPU. Below threshold, heuristic runs in microseconds. Two-phase delivers the right trade-off at any scale.
RNK-04Should
The cross-encoder interface must be pluggable. Swap between local BERT cross-encoder, Cohere Rerank API, and custom providers without changing pipeline code.
§08

Component: Memory Engine

Decay Model

Each conversation turn is scored by an effective score function:

# Effective score formula
effective = importance × recency × freshness + relevance_boost

# Where:
recency          = exp(−decay_rate × age_seconds)
freshness        = exp(−0.01 × seconds_since_last_access)
relevance_boost  = (|query_tokens ∩ turn_tokens| / |query_tokens|) × 0.35
importance       = auto_score(turn_content)  # range: 1.0 – 3.0
MEM-01Must
All decay parameters (decay_rate, freshness_coefficient, relevance_weight, min_importance_threshold) must be configurable per engine instance.
MEM-02Must
Auto-importance scoring must evaluate turn content on three dimensions: content length (log-scaled), domain keyword presence (configurable lexicon), and query token overlap with recent turns.
MEM-03Must
Three-tier deduplication must run before any turn is stored: (1) exact containment check, (2) prefix overlap (first 50% of tokens), (3) token-overlap Jaccard similarity ≥ configurable threshold (default: 0.72).

Persistence

MEM-04Must
Implement a storage backend interface with two concrete implementations: InMemoryBackend (default, session-scoped) and SQLiteBackend (persistent, single-user). The add() and get_weighted() APIs must be identical across backends.
MEM-05Should
Implement a PostgreSQLBackend for multi-user, multi-session enterprise deployments. Must support tenant isolation via session_id and user_id scoping.
MEM-06Could
Implement a VectorMemoryBackend that stores turn embeddings alongside text, enabling semantic recall of past turns (not just lexical overlap) for the relevance_boost calculation.
§09

Component: Semantic Compressor

CMP-01Must
Support three compression strategies: truncate (proportional character cutoff), sentence (greedy sentence-boundary selection), extractive (query-scored sentence selection).
CMP-02Must
All strategies must restore selected sentences to original document order before output. Relevance-rank order produces incoherent context and is explicitly forbidden.
CMP-03Must
The extractive strategy must support two sentence scoring modes: token_overlap (fast, keyword-recall-based) and embedding_similarity (semantic, cosine similarity to query embedding, reusing retrieval model).
Rationale: Token-overlap scoring drops sentences that semantically paraphrase the query. Embedding mode eliminates this blind spot with negligible added latency when the embedding model is already loaded.
CMP-04Must
Report compression_ratio, strategy_used, sentences_selected, and sentences_dropped in the compression result, for inclusion in the context trace.
§10

Component: Token Budget Enforcer

Reservation Order (Non-Negotiable)

# Slot reservation order — must not change
1. system_prompt      # Fixed overhead — reserve first
2. history            # Memory turns — multi-turn coherence depends on this
3. retrieved_docs     # Variable — compressed to fit remaining budget
4. query              # Current user query — always guaranteed space
5. generation_reserve # Output buffer — never consumed by input
BDG-01Must
Token estimation must be configurable: char_heuristic (÷4, default, <1ms), tiktoken (exact, ~5ms per 1k tokens), or model_specific (provider SDK). Production deployments with non-English content or code MUST use tiktoken or equivalent.
BDG-02Must
The budget enforcer must raise a structured BudgetExceededError (not a silent truncation) if the system prompt alone exceeds the total configured budget.
BDG-03Must
Expose a budget_report() method returning per-slot allocation, remaining budget, and overflow risk flag for every build() call. This data must be included in the context trace.
§11

Component: Pipeline Orchestrator

The Orchestrator (ContextEngine class) owns the build(query) method that sequences all components and returns a ContextPacket. 

async def build(self, query: str, session_id: str) -> ContextPacket:
    # 1. Retrieve candidates
    candidates = await self.retriever.retrieve(query, top_k=self.retrieval_k)
    
    # 2. Re-rank
    ranked = await self.reranker.rerank(query, candidates)
    
    # 3. Decay memory
    memory_turns = await self.memory.get_weighted(query=query, session_id=session_id)
    
    # 4. Reserve system + history budget
    budget = TokenBudget(total=self.total_token_budget)
    budget.reserve("system_prompt", self.system_prompt)
    budget.reserve("history", memory_turns)
    budget.reserve("query", query)
    
    # 5. Compress to remaining budget
    remaining = budget.remaining_chars()
    compressed = await self.compressor.compress(ranked, query=query, max_chars=remaining)
    budget.reserve("retrieved_docs", compressed.text)
    
    # 6. Build packet + trace
    return ContextPacket(
        system_prompt=self.system_prompt,
        history=memory_turns,
        retrieved_docs=compressed.text,
        query=query,
        budget_report=budget.report(),
        trace=ContextTrace(...),
    )
ORC-01Must
The build() method must be fully async. All I/O operations (embedding API calls, DB reads, cache lookups) must be awaitable. Synchronous build_sync() wrapper is acceptable as a convenience method.

API Design

Python SDK (Primary Interface)

from context_engine import ContextEngine, EngineConfig

# Initialize with a config profile or explicit parameters
engine = ContextEngine.from_config(EngineConfig(
    profile="conversational",
    total_token_budget=4096,
    compression_strategy="extractive_semantic",
    memory_backend="sqlite",
    memory_db_path="./memory.db",
    reranker_mode="two_phase",
))

# Build a context packet for an LLM call
packet = await engine.build(query="How does memory decay work?", session_id="user-123")

# Compose the prompt using the packet
prompt = assemble_prompt(packet)  # caller's responsibility

# After LLM responds, store the exchange in memory
await engine.memory.add(
    content="How does memory decay work?",
    role="user",
    session_id="user-123",
)

§15

Observability & Evaluation

Context Trace Logging (Required)

Every build() call must produce a ContextTrace object (see §13). Traces must be persisted to a queryable store (SQLite or PostgreSQL, same backend as memory) and accessible via the REST API. No production deployment is complete without traces.

Metrics (Prometheus-Compatible)

  • context_engine_build_latency_seconds (histogram, labeled by mode)
  • context_engine_token_budget_used_ratio (gauge per slot)
  • context_engine_compression_ratio (histogram)
  • context_engine_memory_turns_retained (gauge per session)
  • context_engine_cache_hits_total / cache_misses_total
  • context_engine_reranker_phase (counter: phase 1 vs phase 2 invocations)

Quality Evaluator

A background QualityEvaluator component should run periodic automated evaluation on a held-out query set and track: retrieval NDCG@5 over time, compression semantic similarity (before/after embedding cosine), and memory decay false-positive and false-negative rates. Results must be surfaced in the metrics endpoint and alert on regression.

Risks & Mitigations

RiskProbabilityImpactMitigation
Embedding latency degrades P95 beyond 120ms Medium High LRU cache + async prefetch + GPU batch mode. Fallback to TF-IDF if cache cold on >10 concurrent misses.
Cross-encoder adds unacceptable latency at scale Medium Medium Phase 2 is conditional on document count threshold. Threshold is configurable. Default skips Phase 2 for <20 docs.
Memory decay false positives degrade conversation quality Low High Dedup threshold and decay_rate are configurable. A/B experiments compare configurations on real sessions. Eval suite runs on every release.
tiktoken dependency introduces breaking changes Low Medium Pin tiktoken version. Token estimator is a pluggable interface; char heuristic is always the fallback.
SQLite backend insufficient for high-concurrency production Medium Medium PostgreSQL backend in Phase 2. SQLite is explicitly documented as single-user only.
⚡ Design Principle: Graceful Degradation Over Failure

Every component must define a degraded mode. If cross-encoder unavailable → heuristic re-ranking. If embedding model unavailable → TF-IDF mode with warning. If memory backend down → in-process fallback with warning. Context assembly must always produce a valid (possibly lower-quality) packet. Silent failure is never acceptable.
